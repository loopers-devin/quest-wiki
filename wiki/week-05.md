# Week 5 — 읽기 최적화 (Index · 비정규화 · 캐시)

## 과제 개요

실제 트래픽에서 자주 발생하는 **조회 병목**을 구조적으로 해결한다. 좋아요 순 정렬, 브랜드 필터, 인기 상품 조회 등에서 발생하는 성능 저하를 **인덱스 / 비정규화(또는 MV) / Redis 캐시** 세 축으로 개선하고, 개선 전후를 측정해 비교한다. **3가지 모두 수행하는 것을 권장**한다.

## 🤖 자동 채점 룰 (스킬용)

### 구현 PASS 조건
- ★ 항목 모두 충족 → PASS
- 3개 개선 중 적어도 **인덱스 + 좋아요 구조 개선 + 캐시 1가지 이상**은 반드시 구현
- ★ 중 하나라도 미충족 → **FAIL**

### 구현 FAIL 조건
- 10만 건 데이터 미확보 또는 EXPLAIN 분석 없음 (I1, I3)
- 비정규화/MV 모두 미적용 (S1) — 좋아요 정렬이 여전히 JOIN/GROUP BY로 처리됨
- 좋아요 count 동기화 누락 (S2) — 값이 정적이거나 갱신되지 않음
- Redis 캐시 자체가 없음 (C1+C2 둘 다 누락)
- AS-IS/TO-BE 비교 자료 전무 (M1)

### Rising Talent 신호 (사람 판단용)
- M3(MV) 적용 + 갱신 전략까지 설계
- C6(무효화 전략) 정교함 — 이벤트 기반 invalidation, write-through 등
- 동기화 S4에서 race condition을 분석하고 원자 update / 락 / 이벤트로 해결한 흔적
- 측정 방법론 M2가 구체적 — 부하 패턴, 데이터 분포, 워밍업 분리
- AI1: 설계 점검 결과를 단순 첨부가 아닌 **반영해서 의사결정에 활용**한 흔적

## ✍️ 라이팅 과제 채점

### 제출 형식 (두 가지 중 택1 또는 둘 다)

1. **GitHub Issue (Tech Note)** — 4개 포맷 중 선택
   - 📐 Design Doc / 🪞 Retrospective / ⚔️ Challenge Story / 📊 Benchmark Report
2. **Blog** — 주차 학습/판단 정리, TL;DR 포함

> **참고**: 5주차는 측정 결과를 다루므로 📊 Benchmark Report 포맷이 잘 어울린다.

### PASS 조건 (기본)
- 블로그 또는 Issue 링크 제출 + 접근 가능 + **100자 초과** → PASS

### FAIL 조건 (엄격 적용)
- 미제출 (링크 없음)
- 링크 접근 불가 (404, 권한, 비공개 등)
- 내용이 사실상 없음 (100자 이하, 빈 글)

### 주차별 추가 기준
- (없음 — 기본 규칙만 적용)

### Rising Talent 신호 (사람 판단용)
- 측정 그래프/표가 글에 포함되고 해석이 동반됨
- 트레이드오프 (정합성 vs 성능)에 대한 본인 의견이 명확
- 추천 주제 중 깊이 있게 다룬 경우:
  - 좋아요 순으로 정렬하자 서버가 하염없이 느려졌다
  - 책갈피가 필요한 이유 (Index)
  - 상품 목록에 좋아요 수를 포함하려고 했더니!
  - 캐시 적용 후 DB 호출이 줄었다
  - TTL / 키 설계 / 무효화 결정 기준
  - 정합성과 성능 사이 트레이드오프
