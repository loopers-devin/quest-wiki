# Week 4 — 트랜잭션 · 동시성 제어 · 쿠폰 도메인

## 과제 개요

주문 시 **재고 / 쿠폰 / 주문** 정합성을 트랜잭션으로 보장하고, **낙관적/비관적 락**으로 동시성 이슈(Lost Update)를 제어한다. 새로 추가된 **쿠폰 도메인**(정액/정률, 1회 사용)을 주문 흐름에 통합한다. Application Layer(예: `OrderFacade`)의 트랜잭션 경계 설계가 핵심.

---

### 🎟 Coupon 도메인

| # | 항목 | 필수 | 자동 채점 힌트 |
|---|------|------|--------------|
| C1 | 쿠폰 타입 `FIXED`(정액) / `RATE`(정률) 구분 + 할인 계산 로직 | ★ | `CouponType` enum + `apply(amount)` 도메인 메서드 |
| C2 | 발급된 쿠폰은 사용자 소유 모델로 분리 | ★ | `UserCoupon` / `IssuedCoupon` 등 별도 엔티티 |
| C3 | 발급된 쿠폰은 **단 1회**만 사용 가능 (재사용 불가) | ★ | 상태 전이 메서드 (`use()`) + 이미 USED면 예외 |
| C4 | 상태 `AVAILABLE` / `USED` / `EXPIRED` 구분 + 만료 판단 | ★ | enum + `isExpired(now)` |
| C5 | 최소 주문 금액(`minOrderAmount`) 조건 검증 | ◯ | 정책 검증 메서드 (조건 불충족 시 예외) |

### 🧾 주문 변경 (쿠폰 적용 + 정합성)

| # | 항목 | 필수 | 자동 채점 힌트 |
|---|------|------|--------------|
| O1 | 주문 요청에 `couponId`(NULLABLE) 수용 | ★ | 요청 DTO 필드 + 미지정 허용 |
| O2 | 쿠폰은 주문 1건당 **1장만** 적용 | ★ | 단일 쿠폰만 받는 시그니처 |

| O4 | 주문 성공 시 쿠폰 즉시 `USED` 상태 변경 + 영속화 | ★ | 상태 전이가 동일 트랜잭션 내 |
| O5 | 주문 스냅샷에 **쿠폰 적용 전 금액 / 할인 금액 / 최종 결제 금액** 모두 포함 | ★ | `Order`/`OrderSnapshot` 엔티티 필드 |

## 🤖 자동 채점 룰 (스킬용)

### 구현 PASS 조건
- ★ 항목 모두 충족 → PASS
- ★ 중 하나라도 미충족 → **FAIL**
- ◯는 70% 이상 충족 권장

### 구현 FAIL 조건
- 쿠폰 재사용 방지 미구현 (C3)
- 동시성 테스트 누락 (TC1~3 중 하나라도 없음)
- 트랜잭션 경계가 전혀 없음 (T1, T2)
- 락 전략 자체가 적용되지 않음 (T3)
- 쿠폰 적용 4종 예외 케이스 누락 (O3)

### Rising Talent 신호 (사람 판단용)
- T4(락 선택 근거)가 명료함 — 쿠폰=비관/재고=낙관 등 도메인별 차등 적용
- T5(트랜잭션 분리) — Facade orchestration + 최소 트랜잭션 패턴
- AI1(분석 스킬) 작성 + PR/블로그에서 활용 사례 공유
- 보상 트랜잭션 / 상태 전이 설계 등 한 단계 더 나아간 고민

## ✍️ 라이팅 과제 채점

### 제출 형식 (두 가지 중 택1 또는 둘 다)

1. **GitHub Issue (Tech Note)** — 4개 포맷 중 선택
   - 📐 Design Doc / 🪞 Retrospective / ⚔️ Challenge Story / 📊 Benchmark Report
2. **Blog** — 주차 학습/판단 정리, TL;DR 포함

### PASS 조건 (기본)
- 블로그 또는 Issue 링크 제출 + 접근 가능 + **100자 초과** → PASS

### FAIL 조건 (엄격 적용)
- 미제출 (링크 없음)
- 링크 접근 불가 (404, 권한, 비공개 등)
- 내용이 사실상 없음 (100자 이하, 빈 글)

### 주차별 추가 기준
- (없음 — 기본 규칙만 적용)

### Rising Talent 신호 (사람 판단용)
- "왜 그렇게 판단했는가"가 드러남 (락 선택 근거, 트랜잭션 경계 결정 등)
- 실험/벤치마크 결과 포함 (📊 Benchmark Report 활용)
- 추천 주제 중 깊이 있게 다룬 경우:
  - 트랜잭션이면 안전한 줄 알았다 → rollback 안 되는 경우 실험
  - 쿠폰 사용 처리에 적절한 락은?
  - 스투시 반팔티를 10명이 동시에 주문했을 때
  - 비관적 락 vs 낙관적 락, 선택 기준
  - "우리 상품 좋아요가 한번에 수십개씩 빠져요!" 트러블슈팅
